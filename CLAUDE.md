## Non-obvious gotchas

- **Verification cadence (USER DIRECTIVE 2026-07-10, amended twice 2026-07-11, applies to ALL work in this depot):** the per-task/per-change gate is a **local dual-compile-and-test only** — (a) macOS native (`ctest --preset macos-arm64` + `boot-golden.sh --native`) and (b) rots64 (`ctest --preset linux-x64` + `boot-golden.sh --service rots64 verify`). The second 2026-07-11 amendment moved the i386 container legs (`make test` ctest + tests-Makefile runner + `scripts/boot-golden.sh verify`) to **finalization-only**: qemu-i386 on this Apple-Silicon host costs 60-90+ min per run (and hangs), dwarfing every other leg; the owner accepted paying it once per wave instead of per task. **The old "tolerated qemu SIGSEGV" carve-out for the i386 monolithic runner (`AccountManagement.FormatsCharacterPromptWithLinkedCharacterList`, documented in the USV/Wave-3 exits) is RETIRED as of Wave 4 exit:** it was a stale-object build artifact of the tests Makefile's incomplete hand-written header deps (fixed by `-MMD -MP` auto-deps + a clean rebuild, after which the full monolithic run passes with zero SIGSEGVs), not a qemu artifact — any future monolithic-runner SIGSEGV is a real signal: investigate (start with `make clean`), never tolerate. Do **not** push or trigger the four-platform remote CI per task. **The i386 container battery and the full four-platform remote CI (the only exercise of `windows-msvc`) run only at the wave/branch finalization pass and before any merge.** An i386-only or MSVC-only regression surfacing there is fixed before finalization completes — that deferred risk is accepted in exchange for far cheaper per-task iteration (64-bit-ABI regressions are still caught per-task by the rots64 leg; goldens are still enforced byte-identical per task on macOS + rots64). Goldens are still enforced byte-identical locally on every task. **New-test ASan gate (added 2026-07-10):** any task that ADDS or substantially rewrites a test file must also run that test file once under macOS AddressSanitizer (`cmake --preset macos-arm64` with `-DCMAKE_CXX_FLAGS=-fsanitize=address -DCMAKE_EXE_LINKER_FLAGS=-fsanitize=address`, or an equivalent `-fsanitize=address` build of just the new suite, then run it) — this catches the use-after-free / dangling-fixture-pointer class (e.g. a captured `descriptor.output` that dangles after a reset) that a plain macOS/i386 run passes by luck but that crashes on MSVC's allocator at finalization. Near-zero cost, closes the "new tests aren't Windows-verified until finalization" gap for the memory-error class that actually bit in Phase 4 Wave 2.
- **Docker is no longer mandatory for building/running on this Mac (Phase 2b), and CI is four-platform-required (Phase 3).** The native `macos-arm64` CMake preset (`src/CMakePresets.json`) builds, boots, and passes tests — including the characterization goldens — directly on Apple Silicon: `cd src && cmake --preset macos-arm64 && cmake --build --preset macos-arm64 && ctest --preset macos-arm64`; binary lands at `build/macos-arm64/ageland`, boot-checked with `scripts/boot-golden.sh --native build/macos-arm64/ageland verify`. `windows-msvc` is **green and CI-required as of Phase 3 Task 7** — a hand-rolled, platform-gated `rots_net` socket shim (no third-party networking library; the Task 1 Asio vendoring was reverted per a 2026-07-09 no-third-party pivot) plus chrono/filesystem migrations and MSVC-specific fixes got it there; GoogleTest remains test-only tooling (FetchContent-provisioned on the windows-2022 CI runner only, never shipped in the game binary). Windows verification is build + full ctest (goldens included) on CI — there is no Windows dev machine and no world data for it, so a live Windows **boot** smoke test is explicitly deferred (would need a Windows host plus world data; tracked for a future phase, not silently dropped). The i386 Docker container remains **canonical for the shipping 32-bit binary** and is the legacy-format guard until live server data is confirmed fully migrated off the binary save formats (planned Phase 5) — use it (and the `rots64` sibling service for the 64-bit Linux runtime) whenever you need that exact ABI. See `docs/BUILD.md` and `scripts/rots-docker.sh`. Warnings (`-w` suppresses most) and "deprecated function" notices on build are expected — the game still compiles.
- **STALE since the account-management merge: `-p` is now a port number, not "expect a proxy".** `-p <port>`/`-p<port>` sets the listen port (`comm.cpp:274-290`, default 1024 via `DFLT_PORT`); the flag that means "expect the proxy's 4-byte client-IP header" is now `-x` (`comm.cpp:291-293`, usage line `comm.cpp:413`). Root `make run` runs `./bin/ageland -p 3791` — no `-x` — so it does **not** expect the proxy and plain telnet works directly against port 3791. `scripts/rots-docker.sh boot` boots with neither flag (plain telnet on the default port 1024). Only pass `-x` when a proxy (or `tools/account_smoke.py`, which launches the game with `-x`) is actually sitting in front prepending the IP header — connecting to an `-x` server directly desyncs the very first read.
- **World files and player data are not in this repo.** `lib/world/` comes from a separate repo (https://github.com/Noobinabox/RotS-WorldFiles); `lib/players/`, `lib/plrobjs/`, `lib/exploits/`, and `log/` are git-ignored. Run `cd src && make setup` once to create the runtime directory structure. Never commit anything under these paths.
- **First character created becomes a level-100 Implementor** on a fresh `make setup` install — expected for local dev.
- **Running C/C++ tests** (Google Test): `cd src/tests && make tests`, then run the produced binary at `bin/tests`. The suite has grown to ~1015 tests (1015 total on the i386 container, `rots64`, and macOS native — skip counts differ by platform: 7 skipped in the i386 container vs. 71 skipped on macOS / 73 on rots64 for POSIX/32-bit-fixture-gated cases; Windows count tracked separately in CI, some tests are POSIX-only there), including characterization goldens (`src/tests/goldens/`, `docs/superpowers/goldens/`) that pin existing behavior byte-for-byte — treat a golden diff as a signal to investigate, not to blindly update. Primary verification is still a build + boot smoke test (see `/build-and-smoke`).
- **Toolchain is C++20 on `debian:trixie`/g++ 14.2 containers (Phase 4 Wave 1).** Both `Dockerfile` (i386) and `Dockerfile.x64` build from `debian:trixie`; `src/Makefile`, `src/tests/Makefile`, and `src/CMakeLists.txt` all target C++20. `std::format` is the sanctioned formatting/output-composition target (no third-party `{fmt}` — standing no-third-party-libraries constraint); see `docs/BUILD.md` "Formatting" for the `char[N]`-member decay lesson (`static_cast<const char*>` before handing a fixed-size char array to `std::format` — libc++ vs libstdc++ divergence).
- **Proxy IP header:** `proxy/` (Rust) prepends a 4-byte client IP to the game connection before forwarding. Relevant when touching connection/auth/logging code. `--cloudflare` reads the IP from the `CF-Connecting-IP` header instead.
- **Characterization goldens gate refactors.** `src/tests/goldens/` +
  `scripts/boot-golden.sh` pin behavior under seed 42; run them after any
  combat/persistence/boot change. RNG is owned (`src/rots_rng.*`) — libc
  `rand()`/`random()` are banned.
- **Object/board/mail/pkill/crime/exploit persistence is JSON end-to-end now (Phase 2a).**
  Live saves and loads for these all go through JSON (`objects_json.cpp`, `boards.cpp`,
  `mail.cpp`, `pkill.cpp`, `db.cpp`'s `crime_json`, `exploits_json.cpp`) — nothing on the
  live path writes the old binary struct dumps anymore. The binary decoders documented in
  `docs/data-formats/object-rent-files.md` (now marked LEGACY) still exist, but only as
  one-time migration converters: on first load they detect a pre-migration legacy file,
  decode it, write the JSON equivalent, and rename the legacy file to `.migrated` (never
  deleted outright). Exploit-history conversion is per-character and lazy (triggered by
  that character's next login), so `lib/exploits/` can legitimately still contain many
  un-migrated `.exploits` files at any given time — that is by design, not drift. Don't
  "simplify" a binary decoder by making it whole-struct `memcpy`/`fwrite` again — the
  explicit-offset encoding (where already converted; see `crime_json`) exists specifically
  to survive a future 64-bit rebuild's different `long`/padding sizes. **Player saves are
  different**: only account-native characters save as JSON
  (`account::write_account_character_file`); a character not linked to an account still
  saves through `save_player`'s live line-oriented **text** writer (`db.cpp`, see
  `player-save.md` — not LEGACY, not a migration converter, still written on every save
  for that population). Explicit-offset framing applies to the binary decoders above, not
  to this text format.
